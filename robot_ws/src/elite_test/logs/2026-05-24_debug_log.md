### 10. 串口指令不执行 (核心问题，耗时最长)
- **现象**: Python 发送帧正确，F407 运行正常，但舵机不动
- **调试过程**:
  1. 怀疑 I2C 地址 → 验证 STM32F4 HAL 需 0x80 ✅（无误）
  2. 怀疑 USART 端口 → 改 USART1 ↔ USART2 反复验证
  3. 怀疑 printf 缓冲 → 加 fflush、改 \r\n
  4. 怀疑 DMA 空闲中断 → 注释 ROS2 TX 测试，仍不工作
  5. 怀疑帧解析器 → 发现 ParseVelocityFrame 对非匹配类型返回 1，吞掉关节帧
  - **修复**: 非匹配类型返回 `consumed=0` 让下个解析器处理
  - **文件**: `Hardware/ros2_uart.c`
  6. 仍不工作 → 发现 DMA 空闲中断在 USART2 全双工时不可靠
  - **根因**: `HAL_UARTEx_ReceiveToIdle_DMA` 在 USART2 同时 TX/RX 时空闲检测异常
  - **修复**: 改为 `HAL_UART_Receive_IT` 逐字节中断接收
  - **文件**: `Core/Src/main.c`
  7. ✅ 舵机成功响应

### 11. URDF 文件截断
- **现象**: 编译错误 `unclosed token`
- **修复**: 补回 `</joint>` `</robot>` 闭合标签
- **文件**: `arm_ros2/urdf/arm.urdf`

## 关键调试经验

1. **DMA 空闲中断不可靠**: STM32F4 上 USART2 同时 TX/RX 时，`HAL_UARTEx_ReceiveToIdle_DMA` 的空闲检测可能失效。**逐字节中断接收 (`HAL_UART_Receive_IT`)** 更可靠。

2. **帧解析器不要吞非匹配帧**: 类型不匹配时返回 `consumed=0` 让下个解析器重试，而非 `consumed=total_len`。

3. **ROS2 action 避免在回调中同步等待**: `spin_until_future_complete` 在 executor 回调中会死锁。在 `main()` 中调用（spin 前）或使用回调模式。

4. **USB CDC 缓冲小**: DAPLink VCP 的 USB 缓冲只有 ~512 字节，启动消息会被后续数据快速覆盖。不依赖启动阶段的 printf 调试。

5. **引脚确认是关键**: DAPLink VCP 实际连接的是 USART2 (PA2/PA3)，而非 USART1 (PA9/PA10)。通过分析串口输出中的 ROS2 帧（0xAA 0x01）来确认。
